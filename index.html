OSI Katmanlarına Göre Ağ Problemleri ve Çözümleri

SORUN 1:L1 – Fiziksel Katman Sorunu

🛠️Sorun:
Bu senaryoda, PC ve Server cihazları bir Switch’e bağlanmıştır.
• PC ↔ Switch bağlantısı Copper Straight-through Cable ile yapılmıştır ve bağlantı sorunsuzdur.
• Server ↔ Switch bağlantısı Console Cable (mavi kablo) ile yapılmıştır.
• Console kablosu sadece cihaz yönetimi için kullanılır, veri iletimi sağlamaz.

💬 Belirti:
• Fiziksel bağlantı var gibi görünür.
• LED ışığı kırmızı yanar veya hiç yanmaz.
• ping komutu başarısız olur: Request timed out.

🧪 Teşhis:
• Kablo türü hatalı → fiziksel sinyal taşınamıyor.
• Server veri alamıyor ve cevap veremiyor.
• Sorun tamamen Fiziksel Katman (Layer 1) kaynaklıdır.
• Paket, Switch’e bile ulaşmadan kablo seviyesinde kaybolur.

📁 Hatalı Senaryo Dosyası:
L1 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

Çözüm:
• Console Cable bağlantısı kaldırılır.
• Doğru kablo türü (Copper Straight-through Cable – yeşil renkli) seçilir.
• Server0 FastEthernet0 → Switch0 Fa0/2 bağlantısı yapılır.
• Server’a IP verilir: IP: 192.168.1.2 / Subnet: 255.255.255.0
• ping 192.168.1.2 komutu yeniden çalıştırılır → Reply from 192.168.1.2:.......

📌 Bu sorunda, fiziksel olarak bağlantı yapılmış gibi görünse de kablonun türü hatalı olduğu için veri taşınamamaktadır. Console kablosu veri iletimini desteklemez. Bu nedenle iletişim kurulamamış ve hata alınmıştır. Çözüm, doğru kablo kullanımı ile fiziksel bağlantının uygun hale getirilmesidir. Bu, tipik bir Fiziksel Katman (L1) sorunudur.

📁 Doğru Senaryo Dosyası:
L1 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

SORUN 2:L2 – MAC Flooding ve Port Security

🛠️Sorun:
• Switch portlarına bağlanan cihazların MAC adreslerini otomatik olarak öğrenir.
• Port Security aktif değilse, kötü niyetli kullanıcı tek porttan binlerce sahte MAC adresi gönderebilir.
• Bu durum Switch’in MAC adres tablosunun dolmasına neden olur.

Tablo dolunca:
o Switch hedef MAC adresini bulamaz.
o Tüm portlara broadcast (flood) yapmaya başlar.
o Saldırgan ağdaki tüm trafiği görebilir.

💥 Hatalı Yapılandırma:
• Switch'te hiçbir port güvenliği yok.
• Aynı porta farklı cihazlar bağlanıyor.
• MAC tablosuna birden fazla MAC adresi yazılıyor.
• Switch tüm MAC’leri kabul ediyor (güvensiz davranış).

🧪 Cisco Test Kanıtı:
Switch# show mac address-table
Vlan MAC Address Type Port
1 00D0.AAAA.1111 DYNAMIC Fa0/1
1 00D0.BBBB.2222 DYNAMIC Fa0/1
Aynı porttan 2 farklı MAC geldi = güvenlik açığı

⚠️ Bu Neden Hatalıdır?
• Gerçek ortamda bu davranış veri güvenliğini ihlal eder.
• Saldırganlar başkasına ait trafiği dinleyebilir.
• Layer 2 seviyesinde ciddi bir güvenlik açığıdır.

📁 Hatalı Senaryo Dosyası:
L2 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

✅ Çözüm :
• Switch'e bağlı portlarda Port Security aktif hale getirilir.
• Her port yalnızca tek bir MAC adresine izin verir.
• İlk bağlanan cihazın MAC adresi sticky yöntemiyle otomatik olarak kaydedilir.

• Başka bir cihaz aynı porta bağlanırsa:
o Switch bunu tespit eder.
o Portu otomatik olarak shutdown (kapatır).

📁 Doğru Senaryo Dosyası:
L2 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

🛡️ Ayarlar:
• Port: Fa0/1
• Maksimum MAC: 1
• İhlal Durumu: shutdown
• MAC kaydı: sticky

📄 Cisco CLI Kanıtı:
Switch# show port-security interface fa0/1
Port Security : Enabled
Port Status : Secure-shutdown
Violation Count : 1

Cisco CLI Komutları: 

Switch> enable
Switch# configure terminal
Switch(config)# interface fastEthernet 0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport port-security
Switch(config-if)# switchport port-security maximum 1
Switch(config-if)# switchport port-security violation shutdown
Switch(config-if)# switchport port-security mac-address sticky
Switch(config-if)# end
Switch# write memory

📈 Sonuç:
• İlk gelen MAC adresi otomatik kaydedilir.
• Başka bir MAC gelirse → port kendini kapatır (shutdown).
• Bu yapı sayesinde MAC Flooding saldırıları engellenmiş, sadece yetkili cihazların ağ erişimine izin verilmiştir.
• Veri Bağlantı Katmanı (L2) düzeyinde güçlü bir güvenlik kontrolü sağlanmıştır.

SORUN 3:L3 – Farklı Subnet'te IP Sorunu

🛠️Sorun:
• Ağ katmanı (L3) cihazlara IP adresleri atayarak iletişimi sağlar.
• Aynı yerel ağda bulunan cihazlara, aynı subnet'e ait IP adresleri verilmelidir.
• Eğer farklı subnet’te IP verilirse, cihazlar birbirine ping atamaz.
• Bu projede iki PC aynı Switch’e bağlı olmasına rağmen farklı subnet’e ait IP adresleri ile yapılandırıldığında iletişim kuramadıkları görülmüştür.

Hatalı Senaryo:
Cihaz IP Address Subnet Mask
PC0 192.168.1.10 255.255.255.0
PC1 192.168.2.11 255.255.255.0

• ping sonucu: Request timed out.
• 📌 Sebep: PC'ler farklı ağlara ait olduğu için yönlendirme yapılmaz.
• 📌 Ağ geçidi (gateway) yapılandırılmadığı için iletişim olmaz.

📁 Hatalı Senaryo Dosyası:
L3 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

✅ Çözüm: IP Yapılandırmasının Düzenlenmesi

Cihaz IP Address Subnet Mask
PC0 192.168.1.10 255.255.255.0
PC1 192.168.1.11 255.255.255.0

Test Sonucu (Çözümden Sonra): Reply from 192.168.1.11: bytes=32 time<1ms TTL=128
• Bu sonuç, iki cihazın artık aynı ağda olduğunu ve aralarında başarılı şekilde iletişim kurabildiklerini göstermektedir.

📁 Doğru Senaryo Dosyası:
L3 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

SORUN :L4 – HTTP Port 80 Servisi Kapalı

🛠️Sorun:
Transport Layer, ağ üzerinden iletilen verilerin hedefteki uygulamalara doğru şekilde yönlendirilmesini sağlar. Bu yönlendirme port numaraları aracılığıyla yapılır. Örnek portlar:
• HTTP → Port 80
• HTTPS → Port 443
• FTP → Port 21
Bu senaryoda, istemci (PC0), aynı ağdaki web sunucusuna (Server) HTTP üzerinden erişmek ister. Ancak sunucuda HTTP servisi kapalıdır.

❌ Hatalı Durum (HTTP Kapalı):
Cihaz IP Address Subnet Mask
PC0 192.168.10.10 255.255.255.0
PC1 192.168.10.20 255.255.255.0

🔍 Testler:
• Ping Testi: Başarılı ✅
• Web Erişimi: http://192.168.10.20 → "Server Reset Connection" ❌
📌 Sebep: HTTP servisi kapalı olduğu için port 80 erişimi başarısızdır. L4 düzeyinde port erişim sorunu oluşur.

📁 Hatalı Senaryo Dosyası:
L4 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

✅ Çözüm: HTTP Servisini Açmak
Server’da HTTP servisi aktif edilir.
• Web tarayıcı: http://192.168.10.20 → Cisco Web Sayfası açıldı ✅

📁 Doğru Senaryo Dosyası:
L4 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

📌 Sonuç:
Port 80 erişimi başarıyla sağlanmış, HTTP uygulaması çalıştırılmıştır. Bu, L4 katmanındaki port yönlendirme işlevinin başarıyla gerçekleştiğini gösterir.

SORUN 5:L5 – FTP Servisine Oturum Açılamıyor

🛠️ Sorun:
Session Layer, cihazlar arası oturum açılmasını ve sürdürülmesini sağlar. FTP servisi kapalı olduğunda istemci oturum başlatamaz.

❌ Hatalı Durum :
Cihaz IP Address Subnet Mask
PC0 192.168.50.10 255.255.255.0
PC1 192.168.50.20 255.255.255.0

• Server’da FTP servisi OFF
• ftp 192.168.50.20 → %Error opening ftp://192.168.50.20/ (Timed out) ❌

📁 Hatalı Senaryo Dosyası:
L5 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

✅ Çözümlü Durum:
• Server’da FTP servisi ON
• Kullanıcı: gamze / Şifre: 1234
• ftp bağlantısı başarılı → 230 User logged in

📌 Sonuç:
FTP servisi aktif hale getirildiğinde oturum başarıyla başlatılmış, Session Layer sorunsuz çalışmıştır.

SORUN 6:L6 – Sunum Katmanı Sorunu (index.html biçimsiz)

🛠️ Sorun:
Sunum katmanı, iletilen verinin uygulama katmanına anlaşılabilir ve işlenebilir bir formatta sunulmasından sorumludur. Ancak bu katmandaki bir hata, istemciye (PC) gelen verinin anlamsız ya da biçimsiz görünmesine neden olabilir. Bu durumda istemci cihaz sunucudan veri almasına rağmen, bu veriyi anlamlandıramaz.

🚫 Hatalı Durum:
Sunucu üzerinde HTTP servisi açık olmasına rağmen, varsayılan HTML sayfası düzenlenmemiştir veya boş bir metin ile sunulmaktadır. Kullanıcı http://192.168.10.20 adresine eriştiğinde, boş bir sayfa veya formatlanmamış veriyle karşılaşır. Bu, sunum katmanının görevini yerine getiremediğini gösterir.

📁 Hatalı Senaryo Dosyası:
L6 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

✅ Çözüm :
Sunucu üzerinde HTTP servisi aktif hale getirilmiş, “index.html” dosyası düzgün bir HTML yapısı ile güncellenmiştir. HTML formatı ile biçimlendirilmiş veri, istemci tarafından kolaylıkla yorumlanabilir ve düzgün şekilde görüntülenir. Böylece Sunum Katmanı verinin doğru bir biçimde iletimini sağlar.
🔧 Cisco Packet Tracer Uygulaması:
PC ve Server aynı ağda yapılandırılmıştır (192.168.10.x)
Server'da HTTP servisi aktiftir
“index.html” dosyası aşağıdaki içerikle düzenlenmiştir:

<html> <head><title>Sunum Katmanı Başarılı</title></head> 
<body> 
  <h1>Hoş Geldiniz!</h1> 
  <p>Bu sayfa doğru şekilde sunulmuştur.</p> 
</body> 
</html>



📁 Doğru Senaryo Dosyası:

L6 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

• PC, Web Browser üzerinden http://192.168.10.20 adresine erişmiş ve biçimlendirilmiş web sayfasını başarıyla görmüştür.

SORUN 7:L7 – Application Layer Sorunu: DNS Kayıtlı Değil / Çözümleme Yapılamıyor

🛠️Sorun:Bir istemci cihaz (PC), bir web sitesine (www.ornek.com) erişmeye çalıştığında "Host name unresolved" hatası alıyor. Bu hata, tarayıcının URL’yi karşılık gelen IP adresine çevirememesi durumunda ortaya çıkar. Sorunun temel nedeni DNS servisinin çalışmaması veya istenen alan adı için DNS kaydının olmamasıdır.

🔍 Hatalı Senaryo:
•DNS servisi kapalı ya da aktif değil.
•www.ornek.com alan adı DNS listesinde yer almıyor.
•Tarayıcı üzerinden www.ornek.com girildiğinde "hostname unresolved" hatası alını.r
•Ağ bağlantısı olmasına rağmen web sayfası açılmaz.

📁 Hatalı Senaryo Dosyası:
L7 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

✅ Çözüm :
Server Cihazı Ayarları:
IP Adresi: 192.168.10.10

DNS Servisi: Aktif hale getirildi.
DNS sekmesinden şu kayıt eklendi:
Name: www.ornek.com

Address: 192.168.10.10
HTTP Servisi: Aynı serverda HTTP servisi aktif edildi.
Varsayılan ya da özel bir index.html dosyası bırakıldı.

Client Cihazı (PC) Ayarları:
IP Adresi: 192.168.10.20
DNS Server: 192.168.10.10
Web browser üzerinden http://www.ornek.com adresine erişildi.

Sonuç:
DNS çözümlemesi başarılı oldu.
Web sitesi tarayıcıda açıldı.
DNS sunucusu üzerinden alan adı IP’ye çevrildi ve uygulama katmanı iletişimi başarıyla tamamlandı.

📁 Doğru Senaryo Dosyası:
L7 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

🧩 Açıklama: Bu senaryoda çözümleme işlemi DNS sunucusu sayesinde gerçekleşti. Uygulama katmanındaki bu sorun; kullanıcıların alan adları üzerinden erişim sağlayamamasıyla doğrudan ilişkilidir. Sorunun çözümü için DNS servisinin aktif edilmesi ve ilgili domain kaydının eklenmesi yeterlidir.

SORUN 8: VLAN Hatalı Yapılandırması

🛠️Sorun:
İki bilgisayar (PC0 ve PC1), aynı switch’e bağlı olmalarına rağmen birbirlerine ping atamamaktadır. Kablolama ve IP ayarları doğru olmasına rağmen, “Request timed out” hatası alınmaktadır. Bu durum, veri bağlantı katmanı (L2) kaynaklı bir problemdir.

🧯 Hatanın Teknik Nedeni:
Switch üzerindeki portlar aynı VLAN altında konumlandırılmamıştır.
Örneğin:
PC0 → Fa0/1 → VLAN 10
PC1 → Fa0/2 → VLAN 1 (varsayılan)

📁 Hatalı Senaryo Dosyası:
L8 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)
Bu durum, PC’ler arasında L2 broadcast alanı uyumsuzluğu yaratır. Aynı ağda olsalar bile veri birbirine ulaşamaz.

✅ Çözüm: Tüm portlar aynı VLAN'a alınarak L2 broadcast uyumu sağlanmalıdır. Cisco CLI üzerinden şu komutlar uygulanmıştır:

enable
configure terminal

vlan 10
name ORTAK_VLAN
exit

interface fastethernet0/1
switchport mode access
switchport access vlan 10
exit

interface fastethernet0/2
switchport mode access
switchport access vlan 10
exit

end
write memory

🧪 Test Sonucu:
Her iki bilgisayardan da karşı IP'ye ping atıldığında başarıyla cevap alınmıştır. Sorun çözülmüş, veri bağlantı katmanı senkronize çalışır duruma gelmiştir.

📁 Doğru Senaryo Dosyası:
L8 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

SORUN 9:Firewall (ACL) Hatalı Konfigürasyonu

🛠️Sorun:
Bu senaryoda, router üzerinde tanımlanan yanlış bir Access Control List (ACL) nedeniyle istemciler (Client PC) sunucuya (Server) HTTP üzerinden erişememektedir. Yani kullanıcılar tarayıcıya sunucu IP’sini yazdıklarında sayfa yüklenmez ve “host unreachable” veya timeout hatası alırlar.
Bu sorun hem Network Layer (L3) hem de Application Layer (L7) ile ilgilidir:
•L3: Paket yönlendirme ve ACL filtreleme
•L7: HTTP uygulama hizmetine erişim

❌ Hatalı Yapılandırma:
•Router üzerinde tanımlı ACL (Access List 100), HTTP (port 80) trafiğini engelleyecek şekilde konumlandırılmıştır.
•Bu ACL, ip access-group 100 in komutuyla yanlış arayüze (FastEthernet0/0) uygulanmıştır.

📁 Hatalı Senaryo Dosyası:
L9 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)
•Sunucunun HTTP servisi açık olmasına rağmen, paketler router'dan geçemez.

✅ Çözüm:
•Router CLI arayüzüne girilerek var olan ACL kaldırılmış ve engelleyici kural iptal edilmiştir.
•Gerekirse tüm trafiğe izin veren bir ACL (permit ip any any) tanımlanmıştır.
•HTTP servisi açık olan sunucuya istemciden erişim sağlanmış ve test başarıyla gerçekleştirilmiştir.

🧩Komutlar (Router CLI): 

enable
configure terminal

no access-list 100
interface fastethernet0/0
no ip access-group 100 in
exit
(İsteğe bağlı olarak yeni ACL ile tüm trafiğe izin verilebilir):
access-list 100 permit ip any any
interface fastethernet0/0
ip access-group 100 in
exit

🔍 Test:
Client PC’den Web Browser açılarak http://192.168.20.10 adresine gidildi.
Web sayfası düzgün şekilde yüklendi.
Sorunun firewall (ACL) kaynaklı olduğu doğrulandı.

Bu senaryo, yanlış yapılandırılmış ACL'lerin ağ servislerine nasıl zarar verebileceğini ve uygulama katmanı hizmetlerine (örneğin HTTP) nasıl etki ettiğini gösteren güçlü bir örnektir.

📁 Doğru Senaryo Dosyası:
L9 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

SORUN 10:Tüm Katmanlarda Donanım Uyumsuzluğu

🛠️Sorun: Bir ağ projesinde katmanlar arası uyum eksikliği nedeniyle istemciler (PC'ler) sunucuya erişememektedir. Detaylı analiz sonucunda aşağıdaki katmanlarda sorun tespit edilmiştir:
•L1 (Fiziksel Katman): UTP yerine telefon kablosu (coaxial) kullanılmıştır.
•L2 (Veri Bağlantı Katmanı): MAC adresleri sabitlenmemiştir.
•L3 (Ağ Katmanı): IP adresi yapılandırılmamıştır.
•L4 (Taşıma Katmanı): Uygulamalar port numarası uyumsuzluğu yaşar.
•L7 (Uygulama Katmanı): HTTP servisi devre dışıdır ya da DNS ismi çözülememektedir.
Bu tür katmanlar arası hatalar özellikle büyük ölçekli ağ projelerinde teşhis edilmesi zor ve zaman alıcı problemlere neden olur.

⚙️ Cisco Packet Tracer'da Hatalı Senaryo:
•UTP yerine Phone Cable (telefon kablosu) ile bağlantı yapılmıştır. •Router üzerinde IP yapılandırması eksiktir.
•PC'lerde yanlış gateway verilmiştir.
•Server’da HTTP servisi çalıştırılmamıştır.
•Switch’e bağlı cihazlar farklı VLAN’larda bırakılmıştır.
•DNS çözümlemesi çalışmamaktadır ve "hostname unresolved" hatası alınmıştır.

📁 Hatalı Senaryo Dosyası:
L10 Hatalı Senaryo - Packet Tracer Dosyası (.pkt)

✅ Çözüm:
•Tüm cihazlar uygun UTP kablo ile bağlanmalı.
•Her cihazın IP adresi ve subnet maskeleri doğru girilmeli.
•Router'a varsayılan gateway olarak 192.168.1.1 atanmalı ve aktif hale getirilmeli.
•Switch’te portlar aynı VLAN altında olmalı.
•Server’a DNS ve HTTP servisleri yüklenmeli ve açık olmalıdır.
•Ping ve browser testi ile bağlantı doğrulanmalı.

📁 Doğru Senaryo Dosyası:
L10 Doğru Senaryo - Packet Tracer Dosyası (.pkt)

🎯 Sonuç:
Bu senaryo, tüm OSI katmanlarının birbiriyle uyumlu çalışmasının önemini ortaya koymaktadır. Katmanlar arası hatalar, tek bir katmandan değil; yapıların birlikte çalışamamasından kaynaklanabilir. Bu nedenle, ağ tasarımında hem donanım hem yazılım uyumluluğu test edilmelidir.
